# AIRGAP_CI_GUIDE.md

Гайд по построению CI-пайплайна для Go-проектов в закрытой корпоративной среде (air-gap): откуда сборка тянет сеть, как это перекрыть и какие подходы существуют. Примеры — на базе VictoriaLogs, но применимы к любому Go-проекту.

---

## 1. Что вообще ходит в сеть при сборке Go-проекта

Прежде чем что-то зеркалировать, надо понимать полный список сетевых обращений. «Сборка Go» — это не одно обращение, а до шести независимых каналов:

| Канал | Кто ходит | Куда по умолчанию |
|---|---|---|
| Go-модули (зависимости) | `go build` / `go mod download` | `proxy.golang.org`, fallback `direct` (git+https) |
| Checksum database | `go` при скачивании модулей | `sum.golang.org` |
| Go toolchain | `go` при несовпадении версии из go.mod | через GOPROXY, модуль `golang.org/toolchain` |
| Builder image | `docker pull` | `docker.io` (Docker Hub) |
| Инструменты CI | `go install`, `curl \| sh` (golangci-lint, qtc, govulncheck) | GOPROXY, GitHub, vuln.go.dev |
| Пакеты ОС | `apt`/`apk` при сборке своих образов | зеркала Debian/Alpine |

Плюс инфраструктурный слой: сам Jenkins качает плагины с update-center, агенты качают JDK и т.д. Каждый канал закрывается отдельно — пропустишь один, и «оффлайн»-сборка молча ходит в интернет, пока однажды не упадёт.

## 2. GOPROXY: как Go качает зависимости

### Протокол

`go` не работает с git напрямую (если не сказано иначе) — он ходит по HTTP к module proxy по простому протоколу:

```
GET <proxy>/<module>/@v/list          — список версий
GET <proxy>/<module>/@v/<v>.info      — метаданные версии (JSON)
GET <proxy>/<module>/@v/<v>.mod       — go.mod этой версии
GET <proxy>/<module>/@v/<v>.zip       — архив исходников
GET <proxy>/<module>/@latest          — последняя версия
```

Любой сервер, реализующий эти пять эндпоинтов, — валидный GOPROXY. Именно это реализуют Artifactory (тип репозитория **Go**), Nexus (формат **go proxy**) и self-hosted [Athens](https://github.com/gomods/athens).

### Значения GOPROXY

```bash
GOPROXY=https://proxy.golang.org,direct   # дефолт: прокси, при 404/410 — напрямую через git
GOPROXY=https://artifactory.corp/artifactory/api/go/go-remote   # только внутренний прокси
GOPROXY=off                               # запрет любых скачиваний — сборка падает при попытке
```

Разделители: запятая — «пробуй следующий при 404/410», пайп `|` — «пробуй следующий при любой ошибке, включая сетевую». В закрытом контуре **всегда убирайте `direct`** из цепочки: это fallback на git+https в интернет, который маскирует дыры в зеркале.

### Checksum database (GOSUMDB)

При скачивании модуля `go` сверяет его хэш с публичным журналом `sum.golang.org`. В закрытом контуре два варианта:

1. **Проксировать sumdb** — Artifactory/Athens умеют отдавать его по пути `<proxy>/sumdb/sum.golang.org/...`. Прозрачно, ничего настраивать не надо.
2. **`GOSUMDB=off`** — отключить сверку с публичным журналом. Это **не отключает** проверку целостности: хэши из закоммиченного `go.sum` всё равно сверяются. Теряется только защита от подмены при *первом* добавлении новой зависимости.

Для полностью изолированного контура (Artifactory без egress) — только вариант 2.

### Топологии

- **Прокси с контролируемым egress**: Artifactory стоит в DMZ, remote-репозиторий `go-remote` кэширует `proxy.golang.org`. Разработчики и CI ходят только в Artifactory. Самый частый вариант; «закрытость» обеспечивается тем, что egress есть только у Artifactory.
- **Полный оффлайн**: у Artifactory тоже нет интернета. Тогда зависимости импортируются вручную/по процессу (local-репозиторий, наполняемый из внешнего контура через КСПД/диод). Либо — vendoring, см. ниже.

## 3. vendor/: сборка вообще без прокси

`go mod vendor` складывает исходники всех зависимостей в каталог `vendor/` репозитория. При сборке с `-mod=vendor` Go **не делает ни одного сетевого обращения за модулями** — ни GOPROXY, ни sumdb не нужны.

VictoriaLogs коммитит `vendor/` (обновление — `make vendor-update`), поэтому его CI-окружение:

```bash
GOFLAGS=-mod=vendor   # брать зависимости из vendor/
GOPROXY=off           # любые попытки скачивания = мгновенное падение, а не таймаут
GOSUMDB=off           # сверки с публичным журналом не будет в любом случае
```

Замечания:

- С Go ≥ 1.14 `-mod=vendor` включается автоматически при наличии `vendor/` и актуальном `vendor/modules.txt`, но в CI лучше задать явно — защита от «тихой» деградации в режим `mod`.
- `go mod vendor` кладёт и зависимости тестов, поэтому `go test` тоже работает оффлайн.
- Цена подхода: большие диффы при обновлении зависимостей и раздутый репозиторий. Плата разумная — взамен получаете полностью герметичную и воспроизводимую сборку без единого внешнего сервиса.

### Как гонять vendor-update в закрытом контуре

Обновление зависимостей (`make vendor-update` = `go get -u` + `go mod tidy` + `go mod vendor`) — единственная операция, которой нужен источник модулей: `go get -u` должен *узнавать новые версии* (`@latest`-запросы), без реестра это невозможно. Важно: это разовая maintenance-операция разработчика или отдельной джобы «обновление зависимостей», результат уходит в MR с диффом vendor/ — сборочный CI при этом остаётся герметичным с `GOPROXY=off`.

Три варианта, по убыванию удобства:

**1. Через внутренний go-proxy** (Artifactory/Nexus с контролируемым egress, топология из §2):

```bash
export GOPROXY=https://artifactory.corp/artifactory/api/go/go-remote  # без ,direct
export GOSUMDB=off        # или проксируйте sumdb
export GOTOOLCHAIN=local
unset GOFLAGS             # важно: go get не работает в режиме -mod=vendor
make vendor-update
```

Прокси нужен только для этих редких обновлений — сборка его не касается. Нюанс: `go get -u` может поднять директиву `go` в go.mod; с `GOTOOLCHAIN=local` это честно упадёт, если локальный Go старее — сигнал сначала обновить зеркало builder image (см. §4).

**2. Во внешнем контуре** — если репозиторий синхронизируется с внешним миром (зеркало апстрима, диод, регулярный импорт): vendor-update выполняется там, где есть интернет, коммит с обновлённым vendor/ приезжает внутрь штатным процессом переноса кода. Внутри контура качать ничего не нужно — продолжение той же логики «git clone и есть доставка зависимостей».

**3. Полный оффлайн без прокси** — перенос кэша модулей бандлом (родственно подходу C из §7). Ограничение: discovery версий всё равно требует интернета, поэтому go.mod правится снаружи, а внутри только материализуется vendor/:

```bash
# внешний контур: обновили go.mod, скачали всё
go get -u ./... && go mod tidy -compat=1.26
GOMODCACHE=$PWD/modcache go mod download all
tar czf bundle.tar.gz go.mod go.sum modcache
# закрытый контур: vendor из перенесённого кэша, без сети
GOMODCACHE=$PWD/modcache GOPROXY=off go mod vendor
```

По сути вариант 2 с лишним шагом — имеет смысл, только если снаружи нельзя коммитить.

## 4. GOTOOLCHAIN: скрытое скачивание самого Go

Самая коварная сетевая зависимость. Начиная с Go 1.21 директива `go` в go.mod — это **минимально требуемая версия тулчейна**:

```
go 1.26.4
```

Если в builder image стоит Go 1.26.1, а go.mod требует 1.26.4, то при дефолтном `GOTOOLCHAIN=auto` Go **сам скачает** тулчейн 1.26.4 — как обычный модуль `golang.org/toolchain` через GOPROXY (и проверит через sumdb). В интернет-среде это удобно; в закрытой — сборка либо утечёт в сеть, либо упадёт с невнятной ошибкой скачивания.

Значения:

```bash
GOTOOLCHAIN=auto       # дефолт: скачать нужную версию, если локальная старее
GOTOOLCHAIN=local      # использовать только Go из образа; если он старее go.mod — явная ошибка
GOTOOLCHAIN=go1.26.4   # форсировать конкретную версию (скачает при отсутствии)
```

**В закрытом контуре всегда ставьте `GOTOOLCHAIN=local`.** Ошибка тогда становится честной и понятной:

```
go: go.mod requires go >= 1.26.4 (running go 1.26.1; GOTOOLCHAIN=local)
```

Лечится обновлением builder image, а не сетевым чудом. Следствие: **минорная версия Go в builder image должна быть ≥ директивы `go` в go.mod**, и обновление go.mod (например, после `make vendor-update`) должно сопровождаться обновлением зеркала образа. Это стоит проверять пресабмит-джобой.

Есть и вторая директива — `toolchain go1.X.Y` в go.mod: она задаёт *предпочитаемый* тулчейн (более сильное требование, чем `go`). С `GOTOOLCHAIN=local` обе директивы работают одинаково: «локальный Go должен быть не старее».

## 5. Builder image: что это и как с ним жить

**Builder image** — Docker-образ, в котором выполняется сборка: тулчейн Go + системные зависимости (git, gcc, make). Канонический вариант — официальный `golang:<версия>-<дистрибутив>`:

- `golang:1.26-bookworm` — Debian 12: есть gcc, git, openssl. Нужен для CGO-сборок (`make victoria-logs` использует `CGO_ENABLED=1`).
- `golang:1.26-alpine` — маленький, но musl вместо glibc и **нет gcc** — для CGO придётся доставлять `build-base`.
- Тег `golang:1.26` плавает по патч-версиям — это хорошо (патчи Go содержат security-фиксы), но для воспроизводимости пиньте digest: `golang:1.26-bookworm@sha256:...`.

Builder image ≠ runtime image: собирайте в golang-образе, а в прод кладите бинарь в `scratch`/`distroless`/UBI-micro (multi-stage Dockerfile). Статический бинарь (`CGO_ENABLED=0`, цели `*-pure` в Makefile VictoriaLogs) работает даже в `scratch`.

### Как доставить образ в закрытый контур

1. **Pull-through cache / remote registry** — Harbor proxy-cache, Artifactory docker-remote, `registry:2` в режиме pull-through. Образ прозрачно кэшируется при первом pull. Вариант для топологии «egress есть только у registry».
2. **Импорт вручную** для полного оффлайна:
   ```bash
   docker pull golang:1.26-bookworm
   docker save golang:1.26-bookworm | zstd > golang-1.26-bookworm.tar.zst
   # перенос через диод/носитель, затем:
   zstd -d < golang-1.26-bookworm.tar.zst | docker load
   docker tag golang:1.26-bookworm registry.corp.local/mirror/golang:1.26-bookworm
   docker push registry.corp.local/mirror/golang:1.26-bookworm
   ```
   Либо `skopeo copy docker://docker.io/library/golang:1.26-bookworm docker://registry.corp.local/mirror/golang:1.26-bookworm` — без локального демона.
3. **Свой builder image** поверх зеркала — когда нужны предустановленные инструменты (см. §6):
   ```dockerfile
   FROM registry.corp.local/mirror/golang:1.26-bookworm
   COPY golangci-lint /usr/local/bin/   # версии зафиксированы, скачаны при сборке образа во внешнем контуре
   COPY qtc /usr/local/bin/
   ```

Кто бы ни тянул образ (docker-агент Jenkins делает `docker pull` перед каждым запуском), имя должно указывать на внутренний registry. Настройка `registry-mirrors` в `daemon.json` работает только для Docker Hub-имён; для остальных registry нужны явные внутренние имена.

Не забывайте: образы надо **регулярно обновлять** (патчи Go и Debian) и прогонять через сканер (Trivy/Grype) — закрытый контур не отменяет уязвимостей в тулчейне.

## 6. Инструменты вокруг сборки

Всё, что ставится через `go install` или `curl | sh`, — те же сетевые зависимости:

- **golangci-lint** — `make golangci-lint` в VictoriaLogs качает install-скрипт с интернета. Варианты: запечь бинарь фиксированной версии в builder image (предпочтительно) или положить в internal generic-репозиторий.
- **qtc** (quicktemplate) — нужен только при регенерации шаблонов (`make quicktemplate-gen`), не при сборке. `go install ...@latest` ходит в GOPROXY — в vendored-проекте инструмент в vendor/ не попадает, так что либо запекать в образ, либо не запускать в CI.
- **govulncheck** — качает базу уязвимостей с `vuln.go.dev`. В закрытом контуре нужно зеркало базы (`GOVULNDB=https://mirror.corp/vulndb`) либо исключить из пайплайна и гонять во внешнем контуре.
- **Jenkins-плагины** — сам Jenkins тоже нельзя забывать: плагины ставятся с `updates.jenkins.io`. Стандартное решение — образ контроллера со всеми плагинами, собранный через `jenkins-plugin-cli` во внешнем контуре, плюс зеркало update-center при необходимости.

Правило: **CI-джоба не должна ничего устанавливать в рантайме**. Всё установленное — часть builder image с зафиксированными версиями. Это заодно ускоряет пайплайн и делает его воспроизводимым.

## 7. Три подхода: сравнение

| | A. vendor/ + GOPROXY=off | B. Внутренний go-proxy (Artifactory/Nexus/Athens) | C. Оффлайн-bundle модулей |
|---|---|---|---|
| Сетевые сервисы для сборки | нет | внутренний прокси | нет |
| Новая зависимость | `go mod vendor` + коммит | появляется в прокси автоматически (remote) или по процессу (local) | пересобрать bundle во внешнем контуре |
| Диффы/размер репо | большие | маленькие | маленькие |
| Целостность | код зафиксирован в git | go.sum (+ sumdb, если проксируется) | go.sum |
| Герметичность проверяется | тривиально (`--network=none`) | нужно следить, чтобы не было `direct` | тривиально |
| Кому подходит | monorepo, редкие обновления deps, максимальные требования к воспроизводимости | много Go-проектов в организации, есть Artifactory | среды без repo-менеджеров вообще |

Подход C — это перенос кэша модулей как артефакта:

```bash
# внешний контур:
GOMODCACHE=$PWD/modcache go mod download all && tar czf modcache.tar.gz modcache
# закрытый контур:
tar xzf modcache.tar.gz
GOMODCACHE=$PWD/modcache GOFLAGS=-mod=mod GOPROXY=off go build ./...
```

Подходы комбинируются: например, зависимости — через vendor/ (A), builder image — через Harbor proxy-cache (§5.1), инструменты — запечены в образ (§6). Именно так устроен `Jenkinsfile` в корне этого репозитория.

## 8. Верификация герметичности

«Мы всё замиррорили» — гипотеза, которую надо проверять. Уровни проверки, от простого к параноидальному:

1. **`GOPROXY=off` + `GOTOOLCHAIN=local`** — Go сам становится детектором: любая попытка что-то скачать = мгновенная ошибка сборки с внятным текстом. Ставьте это в env пайплайна всегда, даже если сеть «и так закрыта».

2. **`--network=none`** — запуск сборочного контейнера вообще без сетевого стека:
   ```bash
   docker run --rm --network=none \
     -v "$PWD":/src -w /src -u "$(id -u)" \
     -e HOME=/tmp -e GOFLAGS=-mod=vendor -e GOPROXY=off -e GOTOOLCHAIN=local \
     registry.corp.local/mirror/golang:1.26-bookworm make victoria-logs
   ```
   Это сильнее любого файрвола: ловит и «незаметные» обращения (телеметрию инструментов, DNS-резолвы, скачивания через curl в Makefile). В `Jenkinsfile` этого репозитория `--network=none` вшит в args docker-агента — герметичность подтверждается каждым билдом. Ограничение: стадиям не будут доступны и внутренние сервисы (Artifactory), поэтому приём совместим с подходами A и C, но не B.

3. **Egress-мониторинг** — для подхода B, где сеть до прокси нужна: NetworkPolicy в k8s (deny all egress, кроме Artifactory/registry/git), логирование блокировок на файрволе, алерты на новые направления.

4. **Canary-джоба** — периодическая сборка с чистым кэшем (`GOCACHE`, `GOMODCACHE` — во временный каталог) на чистом workspace: проверяет, что сборка не живёт за счёт случайно накопленного стейта на агенте.

---

## 9. Кэши и скорость

Герметичность не означает медленность:

- **`GOCACHE`** (кэш компиляции) — самый большой выигрыш. Держите его на постоянном volume агента или в директории, переживающей билды. В `Jenkinsfile` репозитория он в `${WORKSPACE}/.gocache` — переживает стадии, но чистится `cleanWs`; для ускорения можно вынести на именованный docker volume.
- **`GOMODCACHE`** — при vendor/ не участвует; при подходе B кэшируйте между билдами, чтобы не гонять модули из Artifactory каждый раз.
- **BuildKit cache mounts** — при сборке Docker-образов: `RUN --mount=type=cache,target=/root/.cache/go-build go build ...`.
- Кэш — это тоже стейт: canary-джоба из §8 защищает от «работает только благодаря кэшу».

## 10. Чек-лист

- [ ] `GOPROXY` — внутренний прокси без `direct`, либо `off` при vendor/bundle
- [ ] `GOSUMDB=off` или проксирование sumdb; `go.sum` закоммичен
- [ ] `GOTOOLCHAIN=local`; версия Go в builder image ≥ директивы `go` в go.mod; процесс обновления образа при бампе go.mod
- [ ] Builder image — из внутреннего registry, пиннинг версии/digest, регулярное обновление и скан
- [ ] Никаких `go install`/`curl | sh` в рантайме джобы — инструменты запечены в образ фиксированных версий
- [ ] Jenkins-плагины/JDK/пакеты ОС — свои зеркала или запечённые образы
- [ ] Герметичность проверяется механически: `GOPROXY=off` в env + `--network=none` (или egress-policy) + canary-джоба с чистым кэшем
- [ ] Runtime-образ отделён от builder image (multi-stage, distroless/scratch)

Рабочий пример всего вышеперечисленного — `Jenkinsfile` в корне репозитория (подход A: vendor/ + `GOPROXY=off` + `--network=none`).
